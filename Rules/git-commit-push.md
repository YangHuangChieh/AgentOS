# Git 提交推送規則

這份文件定義本專案的 Commit 與 Push 流程。

## Commit 訊息格式

Commit 標題必須使用以下前綴之一：

| 前綴 | 用途 |
| --- | --- |
| `[+]` | 新增檔案、功能、資料或規則 |
| `[-]` | 刪除檔案、功能、資料或規則 |
| `[!]` | 修正錯誤、風險、安全問題或破壞性問題 |
| `[*]` | 更新、調整、重構、同步或文件維護 |

格式如下：

```text
[前綴] 繁體中文摘要
```

## 訊息規則

1. Commit 標題與內文一律使用繁體中文。
2. 程式識別字、檔名與必要技術名詞可保留英文。
3. 一個 commit 只使用一個前綴。
4. 若同時包含多種變更，以主要目的決定前綴。
5. 無法明確歸類時，拆成多個 commit。

## 推送規則

1. GitHub 推送完成後的回報內容也必須使用繁體中文。
2. Commit、Push 前，應先列出預計提交的檔案、變更摘要、Commit 訊息、遠端與目標分支。
3. 若實際提交範圍、Commit 訊息、遠端或目標分支改變，原授權失效。
4. Rebase 發生衝突、驗證失敗或需要 Force Push 時，必須停止並重新取得使用者指示。
5. 合併授權不得沿用到後續 Commit 或 Push。
6. 當 AgentOS 有變更時，應自動執行 Commit 並 Push。
7. 每次對話結束後，若有故事、角色、狀態或歷史變更，先同步回對應檔案，再檢查共同資料夾是否需要一併更新並自動 Commit 與 Push。

## 合併授權

使用者可以用一次「同意 Commit 並 Push」授權完整發布流程。

該授權只適用於當次已列出的變更。

## 建議流程

1. 列出預計提交內容。
2. 列出 Commit 訊息。
3. 列出遠端與分支。
4. 取得使用者同意。
5. 進行 Commit 與 Push。
